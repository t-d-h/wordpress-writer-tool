# Plan: Implement Retrying Stage Display for AI Rate Limits and Job Failures

## Overview
The goal is to provide visibility when the backend is automatically retrying an operation, either due to internal rate limiting by AI providers or because a job failed and is being automatically retried. This will be shown on the frontend pipeline as a "Retrying..." state.

## Backend Changes

### 1. Job Auto-Retry Implementation
- Modify `worker/app/workers/tasks.py` to add a generic try/except retry wrapper for tasks.
- If a task fails, increment a `retry_count` (up to a MAX_RETRIES limit like 3).
- If `retry_count < MAX_RETRIES`, update the job status to `retrying` in MongoDB and Redis, log the error, and re-queue the job via `job_service.create_and_queue_job` (or equivalent method to push the existing job payload back to Redis with a delay).
- When the job is re-picked up, it transitions from `retrying` to `running`.
- If max retries are exceeded, the job is marked as `failed` as it currently is.

### 2. AI/Image Service Rate Limit Retries
- In `backend/app/services/ai_service.py`, `_call_ai` currently catches `RateLimitError` and loops for `max_retries` attempts, doing an `asyncio.sleep(delay)`.
- Pass an optional async `on_retry(error, delay)` callback to `_call_ai` (and propagate it down through `research_topic`, `generate_outline`, etc.).
- When `_call_ai` hits a rate limit and is about to sleep, it will call `on_retry`.
- In `tasks.py`, we will pass a lambda or nested async function to these service calls that sets the job status to `"retrying"` (and sets a message like "Rate limit exceeded, retrying in 60s...").
- After the sleep finishes, we need to ensure the status is updated back to `"running"`. We can do this right after the `await asyncio.sleep(delay)` in `_call_ai`.
- Do the same for `backend/app/services/image_service.py` (`generate_thumbnail`) and `backend/app/services/wp_service.py` (`fetch_with_retry` and media uploads if applicable).

### 3. Model/Schema Updates
- The `status` field in MongoDB `jobs_col` and embedded `posts_col.jobs` will support the `"retrying"` string. (This requires no strict schema changes since FastAPI/Pydantic mostly passes strings, but we should verify `post.py` models if they validate the status enum).

## Frontend Changes

### 1. PostView Component (`frontend/src/components/Posts/PostView.jsx`)
- Update the `pipelineSteps.map` rendering logic to handle the `retrying` status.
- Set CSS class to `active` (same as running) or maybe a new `retrying` class with an orange warning color.
- Change the display text:
  ```jsx
  status === 'running' ? 'Running...' :
  status === 'retrying' ? 'Retrying...' :
  status === 'failed' ? 'Failed' : ...
  ```
- Use a spinner or a retry icon (`HiOutlineArrowPath` from `react-icons`) when the status is `retrying`.
- Ensure buttons to trigger jobs remain disabled while the status is `retrying`.

### 2. ProjectDetail Component (`frontend/src/components/Projects/ProjectDetail.jsx`)
- Update `JobStatusBadge` or status configurations to include a `retrying` state.
- Map the `retrying` status to an appropriate icon and class (e.g., `status-retrying`).

## Workflow
1. Update backend AI and Image services to accept and invoke `on_retry` callbacks.
2. Update `tasks.py` to handle job auto-retry on failure and pass rate-limit callbacks to AI services.
3. Verify Pydantic models in `backend/app/models/post.py` to ensure `"retrying"` is a valid status string.
4. Update frontend components `PostView.jsx` and `ProjectDetail.jsx` to render the `"retrying"` state gracefully.
5. Manually trigger a rate limit or a forced failure to test the "Retrying..." UI in the browser.
