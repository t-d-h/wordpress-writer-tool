# Plan: Fix Admin Project Deletion Permission

## Problem
Admin users cannot delete projects they didn't create. The current implementation only allows the project creator to delete/update projects, but admins should have full access to manage all projects.

## Root Cause
In `backend/app/routers/projects.py`:
- Line 219: `delete_project()` checks `current_user["id"] != project["created_by"]`
- Line 174: `update_project()` checks `current_user["id"] != project["created_by"]`
- Line 188: `update_project()` checks `current_user["id"] != wp_site["created_by"]`

These checks don't account for admin users who should have elevated permissions.

## Solution

### 1. Update Project Deletion Permission
**File**: `backend/app/routers/projects.py:210-230`

Change the permission check in `delete_project()` from:
```python
if current_user["id"] != project["created_by"]:
    raise HTTPException(...)
```

To:
```python
if current_user["id"] != project["created_by"] and current_user.get("role") != "admin":
    raise HTTPException(...)
```

## Notes
- The fix follows the existing pattern used in `create_project()` which already checks for admin/editor roles
- Admin role is stored in `current_user["role"]` field
- User ID is stored in `current_user["id"]` field
- This is a comprehensive fix that addresses permission issues across all major resources (projects, posts, wp_sites)
- The `created_by` field will be added to posts and wp_sites during creation to enable ownership tracking
- Existing posts and wp_sites without `created_by` field will need to be handled gracefully (null check in permission logic)
