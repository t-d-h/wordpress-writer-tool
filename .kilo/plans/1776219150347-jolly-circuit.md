### Task 6: Verify BACKEND-05 (cache invalidation)

**Evidence to gather:**
- Check `backend/app/services/wp_cache_service.py` is_cache_stale() method
- Verify method retrieves cached document from wp_posts_cache_col
- Verify method returns True if no cache exists (stale)
- Verify method fetches single post from WordPress API to check for changes
- Verify method compares cached post count with WordPress API total count
- Verify method returns True if counts differ (stale), False if counts match (fresh)
- Verify staleness detection uses get_wp_posts() with per_page=1

**Observable truth:** "Backend implements cache staleness detection by comparing with WordPress API"

**Verification method:** Code inspection + grep commands

### Task 7: Verify BACKEND-06 (cache refresh mechanism)

**Evidence to gather:**
- Check `backend/app/routers/wp_sites.py` for cache refresh endpoint
- Verify endpoint exists: `@router.post("/{site_id}/posts/refresh")`
- Verify endpoint function: `refresh_site_posts_cache()`
- Verify endpoint accepts per_page, page, status, orderby, order parameters
- Verify endpoint calls cache_service.refresh_cache()
- Verify endpoint returns progress information (status, posts_refreshed, total_posts, message)
- Verify endpoint is after get_site_posts endpoint

**Observable truth:** "Backend provides cache refresh endpoint with progress tracking"

**Verification method:** Code inspection + grep commands

### Task 8: Verify cache integration in get_site_posts

**Evidence to gather:**
- Check `backend/app/routers/wp_sites.py` get_site_posts endpoint (lines 102-119)
- Verify WPCacheService is imported and initialized
- Verify cache key is generated using cache_service.get_cache_key()
- Verify hybrid pagination logic:
  - If search parameter is provided: bypass cache, call WordPress API directly
  - If no search parameter:
    - Check cache for existing data
    - If cache exists and is not stale: return cached data
    - If cache is stale or missing: call WordPress API, update cache, return fresh data
- Verify cache is updated after WordPress API call
- Verify error handling for cache operations (try/except blocks)
- Verify logging for cache hits, misses, and staleness detection

**Observable truth:** "Backend integrates cache service with hybrid pagination (cache first, WordPress API fallback)"

**Verification method:** Code inspection + grep commands
