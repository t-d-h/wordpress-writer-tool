# Roadmap: v1.3 Content Quality Improvements

**Milestone**: v1.3
**Status**: Planned
**Last Updated**: 2026-04-16

## Overview

This roadmap defines the phased execution plan for improving content quality in the WordPress Writer Tool. The implementation is divided into 7 phases, each building on the previous one. Focus areas include HTML cleaning, validation (word count and section count), and pipeline integration (research data utilization).

## Risk Mitigation

### Risk: HTML cleaning removes too much content
**Mitigation**: Test with various AI-generated content samples. Review whitelist of allowed tags carefully. Log what HTML cleaning removes for debugging.

### Risk: Word count validation inaccurate for Vietnamese
**Mitigation**: Test with Vietnamese content. Consider using underthesea or pyvi if textstat proves inaccurate. Document known limitations.

### Risk: Section count validation doesn't match user expectations
**Mitigation**: Clarify what counts as a section in documentation. Test with various outline structures. Allow users to override validation warnings.

### Risk: Research data doesn't improve content quality
**Mitigation**: Test with and without research data. Compare content quality manually. Adjust prompts if research data isn't being used effectively.

### Risk: Validation warnings overwhelm users
**Mitigation**: Group related warnings. Provide clear explanations. Allow users to dismiss warnings. Make validation advisory, not blocking.

## Notes

- Three new libraries required: lxml 6.0.4 (HTML sanitization), BeautifulSoup4 4.14.3 (HTML parsing), textstat 0.7.13 (word counting)
- Current implementation has gaps: research data not passed to content generation, no validation for word/section counts, incomplete HTML cleaning
- Validation is advisory, not blocking. Users can publish content that doesn't meet validation criteria.
- HTML cleaning must come before validation to ensure accurate counts.
- Research data utilization enhances content quality by providing context to AI.
- Vietnamese word counting may need underthesea or pyvi if textstat proves inaccurate.
- AI models cannot guarantee exact specifications, hence tolerance-based validation.
