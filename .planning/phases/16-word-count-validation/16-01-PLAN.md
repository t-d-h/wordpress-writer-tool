<objective>
Implement word count validation for posts to ensure content meets quality standards. This involves enhancing the backend to accurately count words in HTML, exposing this functionality via an API, and displaying the validation results in the frontend.
</objective>

<context>
The project currently has a `WordCountService` that uses regular expressions to count words. This plan will improve its accuracy by using a proper HTML parsing library (`BeautifulSoup`) and integrate it into the application workflow.
</context>

  <action>

<task type="auto">
  <name>Task 2: Create Validation Endpoint</name>
  <files>
    - backend/app/routers/posts.py
    - backend/app/models/post.py
    - backend/app/main.py
  </files>
  <action>
    1. In `backend/app/models/post.py`, define Pydantic models for the validation request and response (e.g., `WordCountValidationRequest`, `WordCountValidationResponse`).
    2. In `backend/app/routers/posts.py`, add a new endpoint `POST /api/posts/{post_id}/validate-word-count`.
    3. This endpoint should:
        a. Find the post by `post_id`.
        b. Get the post content.
        c. Use the `WordCountValidator` from `src.validation.word_count` to validate the content. For now, use hardcoded thresholds: `min_words=100`, `max_words=2000`.
        d. Return the validation result using the new Pydantic response model.
  </action>
  <verify>
    <automated>Create a new integration test for the `POST /api/posts/{post_id}/validate-word-count` endpoint.</automated>
  </verify>
  <done>

  </done>
</task>

| T-16-01 | Spoofing | `/api/posts/{post_id}/validate-word-count` | Mitigate | Standard token-based authentication will be handled by the existing framework. No additional work needed. |
| T-16-02 | Tampering | Word count validation result | Accept | The risk of a user tampering with the validation result is low and has minimal impact. The source of truth remains the backend. |
