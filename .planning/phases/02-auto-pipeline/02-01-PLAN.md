<task type="auto">
  <name>Task 2: Modify task handlers to auto-queue next job on success</name>
  <files>worker/app/workers/tasks.py</files>
  <action>
Modify each task handler function to call `queue_next_job` immediately after calling `_update_job_status(job_id, post_id, "completed")`:

1. **run_research**: After completion, queue outline job
   - Call `queue_next_job(post_id, project_id, "outline")`
   - Log: `[PIPELINE] Queued outline job for post {post_id}`

2. **run_outline**: After completion, queue content job
   - Call `queue_next_job(post_id, project_id, "content")`
   - Log: `[PIPELINE] Queued content job for post {post_id}`

3. **run_content**: After completion, queue thumbnail job
   - Call `queue_next_job(post_id, project_id, "thumbnail")`
   - Log: `[PIPELINE] Queued thumbnail job for post {post_id}`

4. **run_thumbnail**: After completion, queue section_images job
   - Call `queue_next_job(post_id, project_id, "section_images")`
   - Log: `[PIPELINE] Queued section_images job for post {post_id}`

5. **run_section_images**: After completion, conditionally queue publish job
   - Read `auto_publish` from post document
   - If `auto_publish` is True, call `queue_next_job(post_id, project_id, "publish")`
   - Log: `[PIPELINE] Auto-publish enabled, queued publish job for post {post_id}`
   - If `auto_publish` is False, log: `[PIPELINE] Auto-publish disabled, stopping pipeline for post {post_id}`

6. **run_publish**: No next job to queue (end of pipeline)

**Critical**: Only queue the next job if the current job completed successfully (status == "completed"). Do NOT queue next job if status is "failed". The existing try/except blocks already handle this by calling `_update_job_status(job_id, post_id, "failed", str(e)` in the except block, so the queue_next_job call in the try block will only execute on success.

**Note**: The `run_publish` task already exists and handles the auto_publish flag correctly (checks both force_publish from job_data and auto_publish from post document). No changes needed to run_publish logic.
</action>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | queue_next_job function | accept | No authentication in system; relies on network isolation. Acceptable for MVP. |
| T-02-02 | Repudiation | Job status updates | accept | No audit logging required for MVP. Acceptable. |
| T-02-03 | Denial of Service | Infinite pipeline loops | mitigate | Pipeline is strictly linear (research→outline→content→thumbnail→section_images→publish) with no cycles. Each job type only queues the next job type once. No risk of infinite loops. |
| T-02-04 | Elevation of Privilege | Job creation | accept | No privilege escalation possible; all jobs run with same worker permissions. Acceptable. |
| T-02-05 | Information Disclosure | Job data in Redis | accept | Redis is internal-only; no sensitive data in job messages beyond post_id and project_id. Acceptable. |

</threat_model>
